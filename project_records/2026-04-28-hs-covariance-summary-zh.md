# 2026-04-28 HS 协变性验证总结

## 目的

这份文档总结两类最容易混淆的 HS 协变性问题，并把这次调试中走过的弯路统一整理出来：

1. 在同一个笛卡尔坐标系下，超胞与原胞如何通过整数扩胞矩阵 `M` 联系起来；
2. 在同一个物理晶格、但笛卡尔坐标轴不同的情况下，两个哈密顿量如何通过真实空间旋转 `Q` 联系起来。

文档重点不是重复中间试错过程，而是给出最终可复用的正确约定、公式、边界条件和排查顺序。

---

## 1. 基本记号与 ABACUS 规范

### 1.1 晶格矩阵的行向量约定

在本项目和 ABACUS `STRU` 中，晶格矢量按“逐行”存储：

```text
A =
[ a1 ]
[ a2 ]
[ a3 ]
```

其中每一行是一个笛卡尔晶格矢量。

若分数坐标写成行向量 `f = (f1, f2, f3)`，则笛卡尔坐标为：

```text
r_cart = f @ A
```

这件事必须先固定，否则后面的 `M`、`Q`、`R` 公式都会左右颠倒。

### 1.2 ABACUS 的实空间哈密顿量约定

ABACUS / pyatb 使用的定义是：

```text
H_{νμ}(R) = <0, ν | H | R, μ>
S_{νμ}(R) = <0, ν | R, μ>
```

也就是说：

- `R` 作用在右边的轨道上；
- 一个块 `(R, atom1, atom2)` 对应的真实空间原子对矢量是

```text
d(R; atom1, atom2) = R @ A + tau_2 - tau_1
```

这里 `tau_1` 和 `tau_2` 分别是 `atom1`、`atom2` 在 `R = 0` 原胞中的笛卡尔位置。

如果把 `tau_1` 和 `tau_2` 写反，后面的 `R'` 映射会整体错号。

### 1.3 ABACUS 只显式存上三角

`data-HR-sparse_SPIN0.csr` 和 `data-SR-sparse_SPIN0.csr` 中，固定一个 `R` 以后，只显式存了该矩阵的上三角。

因此：

- 直接把文件读成一个“只含上三角、下三角为零”的 dense 矩阵，只适合非常有限的场景；
- 如果要恢复完整矩阵，必须结合厄米关系：

```text
H(R) = H(-R)^dagger
S(R) = S(-R)^dagger
```

更准确地说：

- 固定一个 `R`，完整矩阵的上三角来自 `R` 的显式存储；
- 完整矩阵的下三角来自 `-R` 的显式存储再做共轭转置。

对原子块也是同样的道理。若某个 `(R, atom_a, atom_b)` 块处在整块矩阵的下三角侧，不能直接把它当零块，而应切换到它的厄米存储伙伴：

```text
X_eff(R, a, b) = X_store(-R, b, a)^dagger
```

这里 `X` 可以是 `H` 或 `S`。

这一步是这次调试中最关键的坑之一。

## 2. 情况 A：同一个笛卡尔坐标系下的 supercell -> primitive 映射

### 2.1 适用场景

这一类问题满足：

- 源结构和目标结构在同一个笛卡尔坐标系中；
- 不需要对轨道基组做任何真实空间旋转；
- 区别只在于：一个是超胞，一个是原胞，或者两者只是原胞重标记；
- 目标是用整数扩胞矩阵 `M`、原子映射和 `R` 重标记，把两个 HS 联系起来。

这是 `test-abacus-3 -> test-abacus-4` 这类问题的核心。

### 2.2 晶格之间的关系

设：

- 超胞晶格为 `A_sc`
- 原胞晶格为 `A_pr`

在行向量约定下，二者满足：

```text
A_sc = M @ A_pr
```

因此 `M` 的推断应写成：

```text
M_raw = A_sc @ inv(A_pr)
M = round(M_raw)
```

必须检查 `M_raw` 到最近整数矩阵的偏差是否足够小。

若把 `M` 写成 `inv(A_pr) @ A_sc`，就是把左右关系写反了。

### 2.3 单原子映射

设超胞中的原子 `a` 映射到原胞中的原子 `a'`，并且需要一个整数平移 `shift_a` 才能把它拉回到目标原胞内，则有：

```text
tau_a = shift_a @ A_pr + tau'_a'
```

这里：

- `tau_a` 是源结构中原子 `a` 的笛卡尔坐标；
- `tau'_a'` 是目标原胞中原子 `a'` 的笛卡尔坐标；
- `shift_a` 是整数行向量。

### 2.4 原子对 `(R, atom1, atom2)` 的正确映射

对源结构中的原子对，有：

```text
d = R @ A_sc + tau_2 - tau_1
```

代入 `A_sc = M @ A_pr` 和单原子映射：

```text
d
= R @ (M @ A_pr) + (shift_2 @ A_pr + tau'_2) - (shift_1 @ A_pr + tau'_1)
= (R @ M + shift_2 - shift_1) @ A_pr + tau'_2 - tau'_1
```

因此目标原胞中的 `R'` 必须是：

```text
R' = R @ M + shift_2 - shift_1
```

这是这类问题里最核心的公式。

### 2.5 无旋转时，哈密顿量块如何联系

若两边完全在同一个笛卡尔坐标系下，且局域轨道轴、角动量约定、自旋基底都没有变化，则不需要额外的轨道/自旋旋转：

```text
D_a = I
D_b = I
```

这时，几何上对应的块应直接满足：

```text
H_pr(R', a', b') = H_sc(R, a, b)
S_pr(R', a', b') = S_sc(R, a, b)
```

这里唯一要做的是：

- 找对 `(R, a, b) -> (R', a', b')`；
- 正确处理 ABACUS 的上三角存储。

### 2.6 这一类问题里最容易犯的错

#### 错误 1：把 `R` 作用到左边原子

错误写法通常会把真实空间矢量写成：

```text
d_wrong = R @ A + tau_1 - tau_2
```

这会导致：

- `shift_1`、`shift_2` 的位置写反；
- `R'` 公式整体错号；
- 明明几何上对应的块，被错误映射到别的块。

#### 错误 2：把 `M` 作用在错误的一侧

在本问题中，分数坐标是行向量，因此正确式子是：

```text
R' = R @ M + shift_2 - shift_1
```

不是：

```text
R' = M @ R + ...
```

也不是把 `A_sc = A_pr @ M`。

#### 错误 3：认为多个来源块落到同一目标块时应该直接求和

这在“超胞还原成原胞”的验证里是错的。

原因是：

- 超胞里本来就有多个对称/平移等价的原子拷贝；
- 它们映射到同一个原胞目标键，只说明这些来源样本应当彼此一致；
- 在验证阶段应当做“逐样本比较”，而不是“先相加再比较”。

直接相加会把本来正确的等价样本错误放大。

#### 错误 4：忽略边界原子 `1.0 -> 0.0` 的 wrap

ABACUS 会把位于边界上的原子 wrap 回原胞中，例如：

```text
(0, 1, 0) -> (0, 0, 0)
```

如果结构文件、映射程序和 DFT 实际采用的规范不一致，就会出现：

- 原子看起来像多了一个原胞平移；
- `shift` 被误判；
- 后续 `R'` 也跟着错。

因此做原子映射前，应先对结构做统一 wrap。

#### 错误 5：直接把目标下三角块当零块

这是本轮调试里最实质的程序错误之一。

如果一个几何上正确的目标块落在全矩阵下三角，而读取代码只返回“裸上三角 dense”，那么该块会被误读成零块。随后暴力搜索会错误地跑去找另一个“更像”的上三角块。

这会制造出一种假象：

- 几何映射看起来错了；
- 暴力搜索找到的“最佳块”反而更对。

实际上，问题只是读取/比较路径没有切换到厄米存储伙伴。

## 3. 情况 B：同一个物理晶格，但笛卡尔坐标系不同

### 3.1 适用场景

这一类问题满足：

- 源结构和目标结构描述的是同一个 primitive cell；
- 原子之间的真实空间连接关系相同；
- 差别不在 supercell / primitive；
- 差别在于笛卡尔坐标轴不同，或者说整个结构被刚体旋转到了另一个坐标系。

这是 `test-abacus-2/soc -> test-abacus-4` 这类问题的核心。

### 3.2 几何上应该先求真实空间旋转 `Q`

若采用行向量笛卡尔坐标，则正确的几何关系应写成：

```text
r_target ~= r_source @ Q_row
A_target ~= A_source @ Q_row
```

其中：

- `Q_row` 是作用在行向量右侧的真实空间旋转矩阵；
- 若后续代码内部使用列向量旋转，则需转换为

```text
Q_col = Q_row^T
```

这一点必须和 `D` 矩阵构造函数内部约定一致。

### 3.3 `D` 矩阵必须来自真实空间旋转，而不是“看起来像旋转的别的矩阵”

对轨道基组与自旋基组，真正需要的是一个物理的真实空间旋转表示：

```text
D(Q) = D_orbital(Q) ⊗ D_spin(Q)
```

不能把下面这些东西直接拿来当 `D`：

- 扩胞矩阵 `M`
- 晶格基变换矩阵
- 只在分数坐标中成立的坐标变换
- `Q`、`Q^T`、`Q^{-1}` 里随便选一个没核实的版本

这类错误会导致：

- 晶格看似差不多；
- 少量原子点也能拟合上；
- 但 HS 协变性会停留在 `1e-1 ~ 1e0` 量级，而不是 `1e-5` 或更小。

### 3.4 正确的块协变关系

这一类问题中，几何上的 `R` 标号不需要再做 supercell 风格的 `R -> R'` 重标记，而是直接比较同一个 `R`、同一个原子对下的块。

但是块内部需要做轨道+自旋旋转。

若 `D` 被定义为“把源基底系数变换到目标基底系数”的矩阵，则正确关系是：

```text
H_target(R) = D H_source(R) D^dagger
S_target(R) = D S_source(R) D^dagger
```

反向写法是：

```text
H_source(R) = D^dagger H_target(R) D
S_source(R) = D^dagger S_target(R) D
```

这正是 `test-abacus-2/soc -> test-abacus-4` 最终通过验证的关系。

但需要强调的是：

- `D H D^dagger` 还是 `D^dagger H D`，本质上取决于你把 `D` 定义成“正向基底变换”还是“逆向基底变换”；
- 不能脱离 `D` 的代码定义孤立讨论；
- 最可靠的办法是先固定几何约定，再用一个已知应当协变的样例做数值验证。

### 3.5 这一类问题里最容易犯的错

#### 错误 1：只用原子点云拟合旋转

这是本轮调试中最典型的假解来源。

原因是：

- 晶体本身可能有点群对称；
- 只看原子点云时，常常能找到一个“把原子位置也拟合得不错”的假旋转；
- 但那个旋转未必能同时匹配晶格矢量，因此不是整个结构的真实坐标轴变换。

正确做法是联合使用：

- 晶格矢量
- wrap 后的非零原子笛卡尔坐标

一起拟合真实空间旋转。

#### 错误 2：把 `Q`、`Q^T`、`Q^{-1}` 混用

这件事会因为“行向量 vs 列向量”而反复出错。

必须先固定：

- 坐标点是按行向量写还是列向量写；
- 旋转是右乘还是左乘；
- `D` 构造函数期待的输入是 `Q_row` 还是 `Q_col`。

然后整条链路只保留一种约定。

#### 错误 3：只旋转轨道，不旋转自旋

SOC 情况下，`D` 不能只包含轨道角动量表示，还必须包含自旋 `1/2` 的旋转表示。

否则几何看上去完全对，HS 也会系统性不满足协变性。

#### 错误 4：把 k 空间表示矩阵错当成实空间局域基组表示

本轮任务里需要的是“原子局域轨道 + 自旋”的真实空间表示矩阵。

如果误用了只适用于 k 空间小群分析的表示，块协变性就会从定义层面出错。

#### 错误 5：忽略原子在旋转下的置换

真实空间旋转往往不仅改变轨道方向，也会把 `atom_a` 送到另一个原子 `atom_a'`。

因此完整关系不是单纯的块内相似变换，而是：

- 先确定哪个源原子映射到哪个目标原子；
- 再在对应原子块内部施加 `D` 旋转。

## 4. 两类问题的本质区别

### 4.1 情况 A 的核心对象是 `M` 和 `shift`

若两套结构处在同一个笛卡尔坐标系，真正需要回答的是：

- 晶格之间的整数关系是什么；
- 原子在目标原胞中的 home-cell 对应关系是什么；
- 对应的 `R'` 如何写。

这类问题里，几何匹配主要体现为：

```text
R @ A_source + tau_2 - tau_1
=
R' @ A_target + tau'_2 - tau'_1
```

### 4.2 情况 B 的核心对象是 `Q` 和 `D(Q)`

若两套结构描述的是同一个 primitive cell，只是坐标轴不同，则重点不是 `R'`，而是：

- 哪个真实空间旋转把源结构送到目标结构；
- 这个旋转在轨道与自旋基组中的表示矩阵是什么。

这类问题里，几何匹配主要体现为：

```text
r_target ~= r_source @ Q_row
A_target ~= A_source @ Q_row
```

块协变性体现为：

```text
H_target(R) = D H_source(R) D^dagger
```

### 4.3 同时出现时应该怎么做

真实的“结构标准化 + HS 标准化”任务里，这两类问题往往会叠加：

1. 先用 `M`、`shift`、原子映射解决“哪个块对应哪个块”；
2. 再用真实空间旋转 `Q` 解决“对应块内部矩阵元如何旋转”；
3. 最后再统一处理 ABACUS 的上三角存储规范。

也就是说：

- `M` 负责块标签；
- `Q` / `D` 负责块内部基组变换；
- `R <-> -R` 的厄米关系负责把读取路径补完整。

## 5. 推荐的调试顺序

后续再遇到同类问题，建议按下面顺序排查，不要再混着试：

### 第一步：先查几何，不查 HS

先验证：

- `M_raw` 是否足够接近整数矩阵；
- `A_source - M @ A_target` 的误差；
- 单原子映射误差；
- 原子对矢量误差。

如果几何本身不成立，后面一切协变性比较都没有意义。

### 第二步：查坐标轴变换是不是物理旋转

若问题属于不同坐标系情形，必须先验证：

- `det(Q) ≈ 1`
- 晶格误差是否小
- 原子笛卡尔误差是否小

不能只看原子拟合。

### 第三步：查 `D` 的定义和左右次序

在固定 `Q` 以后，再判断：

- `D` 是否同时包含轨道和自旋；
- 代码里使用的是 `D H D^dagger` 还是 `D^dagger H D`；
- 这和 `D` 的定义是否一致。

### 第四步：最后再查 ABACUS 的存储规范

若前面都对，但仍出现大量“某些块等于零、某些块忽然更像别的块”的现象，应优先怀疑：

- 是否直接拿了裸上三角；
- 是否忘了对下三角切换到 `(-R, b, a)^dagger`；
- 是否在同原子块内部遗漏了由 `-R` 提供的下半部分。

---

## 6. 本轮调试的最终经验

本轮最重要的经验可以压缩成四句话：

1. `R` 作用在右边轨道上，因此原子对矢量必须写成 `R @ A + tau_2 - tau_1`。
2. 同坐标系下的 supercell -> primitive 映射，核心公式是 `R' = R @ M + shift_2 - shift_1`。
3. 不同坐标系下的协变性，核心对象不是 `M`，而是真实空间旋转 `Q` 及其轨道+自旋表示 `D(Q)`。
4. ABACUS 只显式存上三角；若不补上厄米伙伴，许多“映射错误”其实只是读取错误。

这四点同时成立，HS 协变性问题才会真正闭合。
